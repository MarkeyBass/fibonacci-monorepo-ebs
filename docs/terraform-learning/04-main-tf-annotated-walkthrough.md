# `main.tf` Annotated Walkthrough (Resource by Resource)

This is a guided "read one resource at a time" walkthrough of `terraform/main.tf`.

Goal:

- You should be able to read the file top-to-bottom and explain **what each block does**, **why it exists**, and **what depends on it**.

How to use this guide:

1. Keep `main.tf` open.
2. Read one section below.
3. Jump to that block in code.
4. Say out loud: "inputs, outputs, and dependencies" for that block.

---

## 0) File preface: Terraform + provider setup

### `terraform { ... }`

What it does:

- Pins Terraform core version to `>= 1.5.0`.
- Pins AWS provider major version to `~> 5.0`.

Why it matters:

- Avoids "works on my machine" drift from major-version behavior changes.
- Gives predictable plan/apply behavior for everyone.

Dependency notes:

- No runtime dependency edges here; this is CLI/provider contract setup.

---

## 1) Provider configuration

### `provider "aws" { region = var.aws_region }`

What it does:

- Configures all AWS API calls to run in your selected region.

Why it matters:

- Region drives endpoint selection and where resources live.
- If region changes, many resources will be replaced in a fresh regional context.

Dependency notes:

- Uses `var.aws_region` from `variables.tf`.

---

## 2) Data sources (lookups, not creations)

Data sources query existing AWS information and make it available to resources.

### `data "aws_caller_identity" "current" {}`

What it does:

- Reads account identity metadata (account ID, caller ARN, etc.).

Why it matters:

- Used in `locals.tf` to build a globally unique fallback S3 bucket name.

### `data "aws_elastic_beanstalk_solution_stack" "docker" { ... }`

What it does:

- Finds the latest Elastic Beanstalk Docker platform matching the regex.

Why it matters:

- You avoid hard-coding an old stack string and can track current platform versions.

### `data "aws_vpc" "default" { default = true }`

What it does:

- Reads the default VPC in this account/region.

Why it matters:

- The whole stack is intentionally "learning speed first" and reuses default networking.

### `data "aws_subnets" "default_vpc_subnets" { filter ... }`

What it does:

- Lists all subnets that belong to that default VPC.

Why it matters:

- RDS and ElastiCache subnet groups need subnet IDs.

Dependency notes:

- Depends on `data.aws_vpc.default.id` through reference.

---

## 3) IAM policy documents (JSON generated by Terraform)

These are still `data` blocks because they construct IAM policy JSON documents, then feed them into role resources.

### `data "aws_iam_policy_document" "eb_ec2_assume_role" { ... }`

What it does:

- Creates trust policy JSON allowing `ec2.amazonaws.com` to assume the role.

Why it matters:

- EB EC2 instances need an instance role.

### `data "aws_iam_policy_document" "eb_service_assume_role" { ... }`

What it does:

- Creates trust policy JSON allowing `elasticbeanstalk.amazonaws.com` to assume the role.

Why it matters:

- Elastic Beanstalk service needs its own service role.

---

## 4) Security group foundation

### `resource "aws_security_group" "multi_docker" { ... }`

What it does:

- Creates one shared SG used by EB app hosts, RDS, and Redis.
- Ingress: allows TCP `5432-6379` from itself (`self = true`).
- Egress: allows all outbound to anywhere.

Why it matters:

- `self = true` means members of the same SG can talk to each other.
- This is a compact way to allow app <-> DB and app <-> Redis connectivity.

Important nuance:

- The ingress port range `5432-6379` is broad. It works for this lab, but in production you usually tighten rules per service/port.

Dependency notes:

- Depends on default VPC lookup via `vpc_id = data.aws_vpc.default.id`.
- Referenced later by RDS, Redis, and EB environment settings.

---

## 5) Deployment artifacts bucket

### `resource "aws_s3_bucket" "deployment_artifacts" { ... }`

What it does:

- Creates S3 bucket for deployment artifacts.
- Uses `local.deployment_bucket_name` (custom or computed fallback).
- Uses `force_destroy` variable to allow easier teardown in labs.

Why it matters:

- EB deployment workflows need a bucket for zipped/source bundle artifacts.

### `resource "aws_s3_bucket_public_access_block" "deployment_artifacts" { ... }`

What it does:

- Enforces "no public access" guardrails on that bucket.

Why it matters:

- Prevents accidental public ACL/policy exposure.

Dependency notes:

- Implicitly depends on bucket ID reference.
- Explicitly listed in EB environment `depends_on` to ensure ordering.

## 6) EB instance role + policy attachments

### `resource "aws_iam_role" "eb_ec2_role" { ... }`

What it does:

- Creates IAM role assumed by EB EC2 instances.

Why it matters:

- App instances need permissions for Elastic Beanstalk managed operations.

### `resource "aws_iam_role_policy_attachment" ...` (3 attachments)

Attached policies:

- `AWSElasticBeanstalkWebTier`
- `AWSElasticBeanstalkWorkerTier`
- `AWSElasticBeanstalkMulticontainerDocker`

Why it matters:

- Gives EB instances permissions expected by the platform/runtime.

### `resource "aws_iam_instance_profile" "eb_ec2_profile" { ... }`

What it does:

- Wraps the EC2 role into an instance profile.

Why it matters:

- EC2 launch configuration references instance profiles, not raw role names.

## 7) EB service role + policy attachments

### `resource "aws_iam_role" "eb_service_role" { ... }`

What it does:

- Creates service role assumed by Elastic Beanstalk control plane.

Why it matters:

- EB service actions (health reporting, managed updates) need dedicated permissions.

### `resource "aws_iam_role_policy_attachment" ...` (2 attachments)

Attached policies:

- `AWSElasticBeanstalkEnhancedHealth`
- `AWSElasticBeanstalkManagedUpdatesCustomerRolePolicy`

Dependency notes:

- These are included in explicit `depends_on` for environment creation stability.

## 8) Database network placement and instance

### `resource "aws_db_subnet_group" "postgres" { ... }`

What it does:

- Defines which subnets RDS can use.

Why it matters:

- RDS must be associated with a DB subnet group in VPC mode.

### `resource "aws_db_instance" "postgres" { ... }`

What it does:

- Creates PostgreSQL instance with variables for size/name/credentials.
- Associates shared SG and subnet group.
- Uses learning-friendly lifecycle flags (`skip_final_snapshot`, `deletion_protection = false`, etc.).

Why it matters:

- This is your persistent relational store for app workloads.

Key settings to understand:

- `publicly_accessible = true`: easier lab access, but usually avoid in production.
- `apply_immediately = true`: faster iteration, but can cause immediate restarts/changes.
- `backup_retention_period = 0`: no automated backups (lab convenience trade-off).

Dependency notes:

- Depends on subnet group and SG references.
- Feeds connection values into EB environment variables.

---

## 9) Redis network placement and replication group

### `resource "aws_elasticache_subnet_group" "redis" { ... }`

What it does:

- Defines subnets available for Redis nodes.

### `resource "aws_elasticache_replication_group" "redis" { ... }`

What it does:

- Creates single-node Redis replication group.
- Uses shared SG and subnet group.
- Enables in-transit encryption with `preferred` mode.

Why it matters:

- Provides cache/session/pub-sub backing for app components.

Key settings to understand:

- `num_cache_clusters = 1`: no replica/high availability.
- `automatic_failover_enabled = false`: learning stack simplicity.
- `at_rest_encryption_enabled = false`: lab-friendly, but production usually enables this.

Dependency notes:

- Later referenced by EB env vars (`REDIS_HOST`, `REDIS_PORT`).

## 11) Elastic Beanstalk environment (the integration hub)

### `resource "aws_elastic_beanstalk_environment" "this" { ... }`

This is where many threads come together.

What it does:

- Creates runnable environment (`tier = "WebServer"`).
- Selects solution stack from data lookup.
- Configures instance type, IAM instance profile, SG, and service role.
- Injects app environment variables for Redis/Postgres connectivity.

Why it matters:

- This is the deploy target your CI pipeline points at.

Nested `setting` blocks explained:

- `aws:elasticbeanstalk:environment / EnvironmentType = SingleInstance`
  - One EC2 instance, low-cost lab mode.
- `aws:autoscaling:launchconfiguration / InstanceType`
  - Controls compute shape.
- `... / IamInstanceProfile`
  - Grants runtime permissions to EC2 host.
- `... / SecurityGroups`
  - Connects host networking to shared SG.
- `... / ServiceRole`
  - Grants EB service-level permissions.
- `aws:elasticbeanstalk:application:environment`
  - Defines runtime env vars your app reads: `REDIS_*`, `PG*`.

Type nuance:

- EB environment values are strings; ports are wrapped in `tostring(...)`.

Explicit dependency strategy:

- `depends_on` ensures IAM policy attachments and bucket access block exist first.
- This reduces intermittent "role not fully ready" and similar creation race issues.

## 12) Dependency graph snapshot (mental model)

Read this as "left creates prerequisites for right":

1. Data lookups -> SG, subnet groups, EB platform selection.
2. IAM policy docs -> IAM roles.
3. IAM roles -> policy attachments -> instance profile / EB service role readiness.
4. SG + subnet groups -> RDS + Redis.
5. RDS + Redis + IAM + platform + app -> EB environment.

If an attribute is referenced, Terraform usually creates producer before consumer.
